## Versal AI-ML Engines GEMM Collection

This repository contains several related GEMM (General Matrix Multiply) designs targeting AMD Versal AI Engine‑ML (AIE‑ML) devices. Each subdirectory is a self‑contained project with its own configuration, build flow, and documentation.

For all provided flows and tiling schemes, the **minimum supported GEMM sizes** are:
- `GEMM_SIZE_A` ≥ 8, `GEMM_SIZE_B` ≥ 8
- `GEMM_SIZE_AB` ≥ 32

The projects are organized so you can start from simple, square GEMM examples and progress to fully end‑to‑end rectangular GEMM with PyTorch comparison on the board.

## 📁 Repository Layout

- `platform_edge_hwemu/`  
  **Reference platform and common hw/hw_emu glue**:
  - Versal edge platform files.
  - Scripts and configuration for running both **hardware (hw)** and **hardware emulation (hw_emu)**.
  - Used as the common platform base for the other GEMM examples.

- `square_GEMM/`  
  **Baseline square matrix multiply design**:
  - Classical \( N \times N \times N \) GEMM on the AIE array.
  - Good starting point if you want to understand the original DSPLIB‑based `matrix_mult_graph` usage and a simpler tiling / cascade scheme.
  - Includes its own `design/`, `Makefile`, and README explaining how to build and run.

- `rectangular_GEMM/`  
  **Rectangular GEMM design** (A×AB×B format), focused on graph structure and tiling:
  - Supports configurations like 8×32×8, 2048×64×8, etc.
  - Uses three explicit sizes:
    - `GEMM_SIZE_A`: rows of A.
    - `GEMM_SIZE_AB`: columns of A / rows of B.
    - `GEMM_SIZE_B`: columns of B.
  - Tile sizes:
    - `DIM_A` and `DIM_B` are derived from `DIM` and split factors.
    - **`DIM_AB` is always** `GEMM_SIZE_AB // CASC_LN_AB` (depends only on cascade length).
  - Includes Python PLIO generators and consistency tools under `design/aie_src/aiesim_data/`.
  - Useful if you want to study the rectangular tiling / cascading and graph mapping without the full end‑to‑end host + ML benchmark flow.

- `rectangular_gemm_end_to_end/`  
  **Complete end‑to‑end rectangular GEMM** flow with host app, DMA kernels, golden models, and ML benchmarking:
  - Starts from host‑side `matrix_A_input.txt` / `matrix_B_input.txt` (or random matrices generated by Python), which are then converted into the on‑board input files/buffers.
  - **A and C in DDR are logical row‑major matrices**:
    - `matA` and `matC` in DDR have the same layout as the Python golden A and A×B.
    - Inside the HLS/DMA kernels, A is **reorganized and replicated** into cascaded streams, and C0/C1 are **reassembled and de‑tiled**, but the numerical values at DDR boundaries are unchanged.
  - AIE graph computes C = A × B.
  - **Pre‑computed B weights (`b*_casc*.txt`)**:
    - `matrix_B_input.txt` can be treated as a fixed **weight matrix**.
    - `plioGen.py` can be run **offline** on the host to generate the cascaded B weight files (`b*_casc*.txt`).
    - These `b*_casc*.txt` files are packaged into the SD‑card image or copied into the board’s DDR filesystem and are consumed directly by the DMA/B path (no re‑tiling of B at runtime).
  - `out_C` DMA kernel **de‑cascades and de‑tiles** the two C streams (C0/C1) back to a single row‑major `matC` buffer in DDR that is numerically equal to the Python golden `matrix_C_golden.txt`.
  - Two comparison modes (see this project’s `config.json` / `SIMPLE_OUT_C`):
    - `SIMPLE_OUT_C = 1`: compare `c.txt` vs `c_golden.txt` (interleaved buffer order).
    - `SIMPLE_OUT_C = 0`: compare `c.txt` vs `matrix_C_golden.txt` (raw row‑major A×B).
  - Offline PyTorch support:
    - `design/ml_wheels/` contains pre‑downloaded PyTorch/torchvision/torchaudio wheels.
    - `setup_ml_environment.sh --offline` installs these on the board **without Internet** and enables on‑board PyTorch GEMM benchmarking.
  - The top‑level `README.md` in this folder documents:
    - DIM/DIM_AB rules,
    - DMA stream ordering,
    - End‑to‑end correctness guarantees,
    - How to run hw_emu / hw and compare against Python / PyTorch.

## 🧱 Common Concepts

Across the projects you will see the same basic parameters:

- **Matrix sizes**
  - `GEMM_SIZE_A`: rows of Matrix A.
  - `GEMM_SIZE_AB`: columns of A / rows of B.
  - `GEMM_SIZE_B`: columns of Matrix B.
  - **Minimum supported sizes** (for the provided flows and tiling):
    - `GEMM_SIZE_A` ≥ 8, `GEMM_SIZE_B` ≥ 8
    - `GEMM_SIZE_AB` ≥ 32

- **Tiling**
  - `DIM_A`: tile size in A dimension, typically `min(DIM, GEMM_SIZE_A / SPLIT_A)`.
  - `DIM_AB`: tile size in AB dimension, **always** `GEMM_SIZE_AB // CASC_LN_AB` in the updated projects.
  - `DIM_B`: tile size in B dimension, typically `min(DIM, GEMM_SIZE_B / SPLIT_B)`.

- **Splits and cascade**
  - `SPLIT_A`: splits rows of A into row blocks.
  - `SPLIT_B`: splits columns of B into column blocks (and C streams).
  - `CASC_LN_AB`: cascade length in AB dimension.

All projects treat A and C as *logical* row‑major matrices in DDR:

- `matA` is the same A that Python uses for its golden model.
- `matC` after `out_C` is equal to Python `A @ B` (within the chosen data type), just tiled and streamed through the AIE in between.

## 🔧 Getting Started

1. **Clone the repo**

```bash
git clone git@github.com:mgrailoo/Versal_AI_ML_Engines_GEMM.git
cd Versal_AI_ML_Engines_GEMM
```

2. **Pick a project**

- For a minimal example: `square_GEMM/`
- For rectangular tiling only: `rectangular_GEMM/`
- For full host + DMA + golden comparison + PyTorch: `rectangular_gemm_end_to_end/`

3. **Follow the project‑local README**

Each subproject has its own `README.md` with:

- Required Vitis/Vivado versions.
- How to edit `design/design_configs/config.json` (GEMM sizes, DIM, splits).
- Build and run commands (`make run TARGET=hw_emu`, `make run TARGET=hw`, `make report_metrics`, etc.).
- How to generate and interpret golden files and output comparisons.
